프로젝트 내비게이션 상태 표현 — 컨셉 비교

현재 구현은 “현재 페이지”와 “다음 추천 단계”를 모두 같은 파란색으로 표시해 구분이 어렵습니다. 아래 7가지 컨셉은 색상 외에 타이포그래피·아이콘·라벨·모양을 결합해 같은 상태를 더 명확하게 전달할 수 있는지 비교하기 위한 것입니다. 모든 컨셉은 동일한 상태로 동시에 업데이트됩니다.

컨트롤을 변경하면 아래 7개 컨셉이 동시에 갱신됩니다.

Concept A — 타이포그래피 & 아이콘 위계

기존 구조 유지

현재의 가로 메뉴 구조를 거의 그대로 두고, 굵기·밑줄·아이콘·배지만으로 다섯 가지 상태를 구분합니다. 그레이스케일로 봐도 상태를 구분할 수 있어야 합니다.

    Concept B — 상태 라벨

    라벨 태그

    각 메뉴 항목 위에 완료·현재·다음·선택·개요 같은 작은 의미 라벨을 붙입니다. 명확함은 커지지만, 시각적 무게도 함께 커지므로 그 비용이 정당한지 판단하는 것이 목적입니다.

      Concept C — 스텝 인디케이터 + 분리 배치

      필수/유틸리티 분리

      필수 단계(아이디어→첫 버전→테스트)는 진행 스텝으로 표현하고, 리스크·개요는 별도 줄에 다른 시각 언어로 배치합니다. 리스크가 개요보다 먼저 완료돼야 한다는 인상을 주지 않아야 합니다.

          Concept D — 내비게이션 + 진행 요약

          보조 정보 한 줄

          메뉴 자체는 간결하게 유지하고, 그 아래 한 줄로 전체 진행 상황을 요약합니다. 메뉴 안에서도 현재·다음이 구분되는지 함께 확인합니다.

            Concept E — 내비게이션 + 다음 단계 액션

            별도 CTA 패널

            메뉴는 위치·상태 전달에만 집중하고, 다음 단계 제안은 그 아래 별도 패널로 분리합니다. 광고처럼 보이지 않고 워크스페이스의 일부처럼 느껴져야 합니다.

              Concept F — 프로그레시브 타임라인

              체크아웃과 다른 느낌

              타임라인 형태를 탐색하되, 계정 개설이나 결제 단계처럼 보이지 않고 AI 워크스페이스의 진행 흐름처럼 느껴지도록 절제합니다. 좁은 화면에서는 세로로 전환됩니다.

              Concept G — 권장 하이브리드

              최종 권장안

              앞선 컨셉들의 강점을 조합합니다: 현재 페이지가 가장 강한 강조를 받고, 완료는 체크 아이콘, 다음은 화살표+배지로 파란색에 의존하지 않고 구분되며, 필수 단계는 하나의 그룹으로 묶이고, 리스크는 선택 도구로, 개요는 워크플로 단계가 아닌 상시 접근 가능한 목적지로 표현됩니다.

              프로젝트

              방구 대신 끼어주는 앱

              • 묶인 필수 그룹 — 아이디어·첫 버전·테스트가 하나의 캡슐 안에 있어 “이것이 순서가 있는 워크플로”임을 형태로 전달합니다.
              • 현재 = 배경 필 + 밑줄 + 굵은 글씨 — 세 가지 신호가 겹치므로 그레이스케일에서도, 색약 사용자에게도 가장 강하게 읽힙니다.
              • 다음 = 화살표 + 앰버 배지 — 파란색이 아닌 별도 색상과 아이콘 조합이라 “현재”와 절대 혼동되지 않습니다.
              • 리스크 = 그룹 밖 + 다른 모양 — 같은 캡슐에 넣지 않고 외곽선 알약 모양으로 분리해 “완료해야 할 단계가 아님”을 형태로 전달합니다.
              • 개요 = 구분선 뒤 상시 링크 — 번호도 체크마크도 없이, 항상 존재하는 목적지로 취급됩니다.

              비교 매트릭스

              동일한 상태값(완료: 아이디어 · 현재: 첫 버전 · 다음: 테스트)에서 7개 컨셉을 평가했습니다. 모든 컨셉이 우수하다고 말하지 않습니다 — 실제 트레이드오프를 그대로 남겼습니다.

              컨셉 현재 위치 인지 다음 단계 인지 완료 상태 인지 전체 흐름 이해 시각적 복잡도 기존 UI 적용 난이도 모바일 확장성 단계 증가 대응 접근성

              권장안

              권장 컨셉: G — 권장 하이브리드. 파란색 하나에 의존하던 기존 문제를 형태(그룹핑)·아이콘(체크/화살표)·위치(구분선)의 조합으로 해결하면서, 헤더에 들어갈 만큼 컴팩트합니다. 필수 단계가 늘어나도(예: 시장 조사, 페르소나 추가) 캡슐 안에 항목만 추가하면 되므로 구조가 깨지지 않습니다.

              유지해야 할 요소

              • 현재 페이지에 3가지 신호를 겹치는 것 (배경 필 + 밑줄 + 굵기) — 색 하나에 의존하지 않는 핵심 장치
              • 완료 = 체크 아이콘 (Concept A/C/F 공통으로 가장 즉각적으로 읽힘)
              • 다음 = 화살표 + 다른 색상 배지 (파란색과 절대 겹치지 않음)
              • 리스크·개요를 필수 워크플로 캡슐 밖에 배치 (Concept C의 분리 원칙)

              채택하지 않을 요소

              • Concept B의 항목별 라벨 칩 — 다섯 항목 모두에 라벨을 얹으면 다섯 개의 새로운 텍스트 블록이 생겨 오히려 스캔 속도를 늦춤
              • Concept E의 상시 노출 CTA 패널 — 유용하지만 헤더 높이를 늘려 “컴팩트한 헤더” 제약과 충돌; 온보딩 첫 진입 등 좁은 맥락에서만 조건부로 고려
              • Concept F의 원형 타임라인 마커를 기본값으로 — 계정 개설/체크아웃 위저드 느낌에 가장 가까워, 채택 시 절제된 크기로만

              현재 페이지와 다음 단계가 같은 색을 공유해야 하는가?

              아니요. 이번 프로토타입의 근본 원인입니다. 현재는 accent 블루 하나로, 다음은 별도의 앰버(호박색) 계열로 분리하고, 형태(밑줄+필 vs. 화살표+배지)까지 함께 다르게 해야 합니다.

              완료 단계에 체크 아이콘을 써야 하는가?

              예. 텍스트 대비 스캔 속도가 가장 빠르고, 그레이스케일·색약 환경에서도 형태로 구분됩니다. 다만 완료 텍스트 자체는 필수 그룹 안에서 여전히 중간 명도를 유지해 현재 페이지보다 시각적으로 가라앉아야 합니다.

              리스크는 워크플로 행(row)에 남아야 하는가?

              시각적으로는 남되, 형태로 분리돼야 합니다 (다른 모양, 그룹 캡슐 밖). 완전히 다른 줄로 옮기면 “프로젝트 안의 도구”라는 정체성이 약해지고 별도 설정 메뉴처럼 보일 위험이 있습니다 — 이는 UX 판단이며, 최종 배치는 프로덕트 팀의 결정 사항입니다.

              개요는 처음에, 끝에, 아니면 별도 유틸리티 그룹에 있어야 하는가?

              별도 그룹 끝쪽을 권장합니다 — 워크플로 캡슐과 시각적으로 분리하되, 여전히 프로젝트 내비게이션의 일부처럼 보이는 위치. “처음”에 두면 필수 순서의 0번째 단계처럼 오인될 수 있습니다.

              별도의 다음 단계 CTA가 꼭 필요한가?

              필수는 아닙니다. Concept G(메뉴 자체의 명확성)만으로 “다음이 무엇인지”는 충분히 전달됩니다. CTA 패널(Concept E)은 온보딩 직후처럼 설명이 더 필요한 맥락에서 추가하는 강화 옵션으로 남겨두는 것을 제안합니다.

              1차 구현에 반드시 필요한 상태 구분은 무엇인가?

              현재·완료·다음 세 가지입니다. future(향후)와 optional(선택)은 “강조되지 않음”이라는 동일한 시각적 결로 묶어도 무방합니다 — 다섯 가지를 모두 똑같이 강하게 구분하려 하면 오히려 위계가 무너집니다.

              잠재적 리스크

              • 완료 체크 + 필 그룹 + 다음 배지가 좁은 모바일 폭에서 줄바꿈되면 위계가 흐트러질 수 있음 — 반응형 검증 필요
              • 다음 단계 강조가 “시스템이 순서를 강요한다”는 인상을 줄 수 있음 — 이전 리뷰(ADR-0020)에서 이미 지적된 위험과 동일한 맥락이므로, 시각적 강조가 실제 잠금(lock)으로 오인되지 않도록 카피와 상호작용으로 뒷받침해야 함
              • 앰버 색상이 warning(경고) 색상과 근접해, 다음 단계가 “문제가 있다”는 오해를 줄 가능성 — 사용자 테스트로 확인 필요

              사용자 테스트로 확인해야 할 것

              • 5초 노출 후 “지금 어느 페이지에 있다고 생각하는지” 회상 테스트 (현재 vs. 다음 혼동 여부)
              • 처음 보는 사용자에게 “다음에 무엇을 해야 할지” 질문 — 화살표+배지만으로 충분한지, 텍스트 라벨(다음)이 꼭 필요한지
              • 리스크가 완료해야 할 필수 단계로 오인되는 비율
              • 색맹 시뮬레이션 환경에서의 완료/현재/다음 구분 정확도

              제안 구현 순서

              1. 현재 페이지 표시를 3중 신호(필 배경 + 밑줄 + 굵기)로 먼저 교체 — 가장 시급한 버그(현재/다음 혼동) 해결
              2. 다음 단계를 별도 색상(앰버)과 화살표 아이콘으로 분리 — 여전히 색상 하나에만 의존하지 않도록 배지 텍스트(다음) 동반
              3. 완료 상태에 체크 아이콘 추가
              4. 필수 3단계를 시각적으로 그룹핑(캡슐/구분선)하고 리스크·개요를 그룹 밖으로 분리
              5. 반응형 줄바꿈 및 접근성(aria-current, 포커스, 라이브 리전) 검증
              6. 사용자 테스트 후 CTA 패널(Concept E) 추가 여부를 별도 결정

              가정 — 이 프로토타입은 상태 전달의 명확성만 다룹니다. 향후 단계가 숨김·비활성화·잠금 중 무엇으로 처리될지는 별도의 프로덕트 정책 결정이며, 여기서 시각적으로 결정하지 않았습니다. 모든 컨셉은 요청대로 다섯 항목을 항상 클릭 가능하게 유지합니다.
